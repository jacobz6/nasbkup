# NAS Backup 恢复功能需求对齐文档 (ALIGNMENT)

## 1. 业务目标

完成 NAS 云端备份系统从"备份"到"恢复"的完整闭环，解决三类核心恢复场景：
1. **本地文件丢失/损坏恢复**：NAS 系统完好，部分文件误删/损坏，从云端恢复指定文件/目录
2. **整台 NAS 丢失灾难恢复**：NAS 设备故障/丢失，需要在新环境重建备份系统并恢复数据
3. **全新 NAS 全盘迁移**：购买新 NAS 设备，将云端全部备份数据完整恢复到新设备

## 4. 场景定义与验收标准

### 场景 A：部分文件丢失恢复（最常用）
**触发条件**：用户误删了某些文件/目录，NAS 系统正常运行，备份数据库完好
**操作流程**：
1. 用户在 Web UI 进入"恢复"页面
2. 浏览或搜索已备份的文件/目录
3. 勾选需要恢复的文件/目录
4. 选择目标目录（默认为原路径，或指定恢复目录）
5. 可选：选择恢复到特定历史版本
6. 点击"开始恢复"，实时查看进度
7. 恢复完成后查看结果（成功/失败文件列表）

**验收标准**：
- 能正确列出所有 active 状态的已备份文件
- 勾选文件后能正确触发恢复
- 恢复进度实时更新（已恢复/总数/已恢复大小/速度/预计剩余时间）
- 恢复完成后文件能正确解密、解压、哈希校验通过
- 原文件存在时给出冲突处理选项

### 场景 B：整台 NAS 丢失灾难恢复
**触发条件**：NAS 硬件故障/丢失/系统彻底损坏，本地数据库和 master.key 均丢失
**前置必备文件（必须在云端之外安全保存）**：
1. `master.key` - 主密钥文件（用于解密所有备份数据，丢失则数据永远无法恢复）
2. `rclone.conf` - rclone 配置文件（包含 OSS 访问凭证和 crypt 密码）
3. `config.yaml` - 应用配置文件（可选，可重新配置）
4. 原备份目录结构记录（可选，便于恢复到原路径）

**操作流程**：
1. 在新设备/NAS 上安装 nas-backup 系统
2. 放置必备文件到正确位置（master.key、rclone.conf）
3. 重新配置 config.yaml（指向相同 OSS bucket 和 crypt 配置）
4. 启动系统，系统自动扫描 OSS 重建元数据（或从备份恢复数据库）
5. 进入 Web UI 验证云端文件可见
6. 执行全盘恢复或选择性恢复

**验收标准**：
- 文档中明确列出必备文件清单和保存建议
- 文档中提供分步灾难恢复操作指南
- 新部署的系统能连接到原有 OSS 备份仓库
- 能列出之前备份的所有文件（需处理数据库重建问题）

### 场景 C：全新 NAS 全盘迁移
**触发条件**：旧 NAS 仍可用，购买新 NAS，要将所有备份数据恢复到新设备
**与场景 B 的区别**：可以从旧 NAS 导出数据库和配置文件，不需要完全重建
**操作流程**：
1. 在新 NAS 安装 nas-backup
2. 从旧 NAS 复制：master.key、rclone.conf、config.yaml、nas-backup.db
3. 启动新 NAS 上的备份服务
4. 在 Web UI 点击"一键全盘恢复"
5. 选择目标目录（新 NAS 的数据目录）
6. 等待全量恢复完成
7. 验证文件完整性

**验收标准**：
- Web UI 提供"一键全盘恢复"按钮
- 能计算全盘恢复的总文件数和总大小
- 恢复完成后所有文件可正常访问、哈希校验通过

## 5. 关键决策点与歧义点

### 决策点 1：灾难恢复时的数据库重建策略
当前系统依赖本地 SQLite 数据库记录文件元数据和哈希索引。如果整台 NAS 丢失，数据库也会丢失，仅有 OSS 上的加密数据对象。

**方案 A**：要求用户定期备份本地数据库文件（nas-backup.db）到安全位置，灾难恢复时直接放回
- 优点：实现简单，恢复速度快，保留完整历史
- 缺点：用户必须额外备份数据库，否则无法恢复文件列表；数据库可能落后于最新备份

**方案 B**：增加"OSS 元数据重建"功能，系统启动时扫描 OSS 上的所有对象重建数据库
- 优点：不需要用户备份数据库，真正的无状态灾难恢复
- 缺点：OSS 对象列表扫描可能很慢（大量对象时），需要在对象元数据中存储足够信息来重建路径和哈希关系
- 前提：当前备份时是否在 OSS 对象元数据中存储了路径、哈希、压缩/加密信息？需要调研

**我的初步倾向**：方案 A（文档中明确要求用户备份 master.key + rclone.conf + nas-backup.db）+ 方案 B 作为后续增强。原因：当前系统未在 OSS 对象 metadata 中存路径信息（路径存在本地 DB 的 backup_files.storage_key 映射），方案 B 实现复杂度高，建议在本期先做方案 A，并在文档中强调"必备三件套"的异地备份重要性。

### 决策点 2：恢复目标目录选择方式
**方案 A**：仅允许恢复到预先配置的"恢复暂存目录"（白名单）
- 优点：安全，无路径遍历风险
- 缺点：不灵活，恢复后用户需要手动移动文件

**方案 B**：允许用户选择任意目录，但限制在配置的"允许恢复根目录"范围内
- 优点：可以直接恢复到原路径，使用方便
- 缺点：有路径遍历风险，需要严格校验

**方案 C**：提供两个选项："恢复到原路径"（直接覆盖原位置）和"恢复到暂存目录"（恢复到指定安全目录供用户检查后再移动）
- 优点：兼顾安全和便利
- 缺点：UI 稍复杂

**我的初步倾向**：方案 C，日常误删恢复可直接恢复到原路径（方便），灾难恢复/迁移恢复到指定目录（安全）。

### 决策点 3：恢复进度实时推送机制
当前备份进度通过 SSE (`/api/backup/progress`) 推送，使用全局 progress 结构体。

**方案 A**：复用现有 SSE 基础设施，增加 restore 事件类型
- 优点：无需新建端点，前端复用 EventSource 逻辑
- 缺点：备份和恢复同时进行时进度事件会混淆

**方案 B**：新建独立的 `/api/restore/progress` SSE 端点，按 jobID 区分
- 优点：职责分离，支持多个恢复任务查询
- 缺点：需要新增代码，复用率低

**我的初步倾向**：方案 B，因为恢复可能运行数小时（解冻+下载），用户可能在恢复期间启动备份（虽然应该加互斥），独立端点更清晰。但由于当前系统设计上备份和恢复应该互斥（都有并发锁），方案 A 也可接受，实现更快。

### 决策点 4：Web UI 文件浏览方式
**方案 A**：树形目录浏览器（类似文件管理器，逐级展开）
- 优点：直观，适合按目录浏览和选择
- 缺点：实现复杂，需要目录层级 API

**方案 B**：平铺列表 + 搜索过滤 + 路径筛选（类似当前 Content 页面）
- 优点：实现简单，复用现有 DataTable 组件；支持搜索、排序、分页
- 缺点：目录级选择需要额外处理（选择目录 = 递归选中该目录下所有文件）

**我的初步倾向**：方案 B（平铺列表+搜索+路径前缀筛选），参照现有 Content.tsx 的风格实现，加快开发速度。同时提供"全选"和"按目录选择"的快捷操作。

## 6. 不在本期范围内的功能（Out of Scope）

- 用户认证/权限系统（整个系统目前无认证，恢复模块不单独引入）
- 恢复到不同 NAS 的网络直传（恢复目标必须是运行 nas-backup 服务的本机）
- 文件版本历史对比（仅提供版本选择，不做 diff 对比）
- 恢复前文件预览（加密文件无法预览）
- 移动端适配（当前系统也未做移动端专门适配）
- 定时恢复计划（恢复是手动触发操作）

## 7. 假设条件

1. 用户会按照文档要求安全备份 `master.key`、`rclone.conf`、`nas-backup.db` 三个关键文件
2. 恢复操作在 nas-backup 服务运行的同一台机器上执行，恢复目标目录是本机可访问的目录
3. OSS 访问凭证有效，网络连接正常
4. ColdArchive 解冻费用和等待时间用户已知晓并接受
5. 恢复期间系统不会同时执行备份操作（或通过现有互斥锁机制防止并发）
